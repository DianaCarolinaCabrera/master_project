Guía completa de uso, parámetros, capacidades y limitaciones  ·  Versión 1.0

¿Qué es esta herramienta?

El ML Arch Recommender es una aplicación web que ayuda a data scientists, ingenieros de ML y arquitectos cloud a decidir cuál es la mejor forma de desplegar un modelo de Machine Learning en AWS, según las características específicas de su caso de uso.

La herramienta evalúa automáticamente 5 patrones de despliegue (Serverless, Batch, Streaming, Containers, SageMaker) usando 5 criterios ponderados y devuelve una recomendación justificada con nivel de confianza, análisis de costos y guía de implementación.

En resumen: Responde la pregunta “¿Dónde y cómo despliego mi modelo en AWS?” en menos de 60 segundos, con criterio técnico y económico.

Requisitos

Para usar la aplicación web (recomendado)

  • Navegador web moderno: Chrome 90+, Firefox 88+, Edge 90+, Safari 14+
  • Conexión a internet (solo para cargar la página; el motor funciona sin conexión)
  • No se requiere instalación, registro ni cuenta

Cómo acceder

Opción A — Aplicación web (sin instalación)

Abre el enlace HTTPS proporcionado en tu navegador. La aplicación carga en ~3 segundos.

URL de producción: La URL HTTPS se genera automáticamente al desplegar en Streamlit Cloud. Ver la documentación de despliegue para obtenerla.

Opción B — Ejecución local

git clone <repositorio>
cd project_root
poetry install
poetry run streamlit run app.py

La aplicación abre automáticamente en http://localhost:8501.

Paso 1 — Configura los parámetros del modelo

El panel lateral izquierdo contiene el formulario de configuración. Tienes dos formas de llenarlo:

A

Manualmente — llena cada campo

Ingresa los valores directamente en los campos del panel lateral: descripción del caso, tipo de modelo, latencia requerida, frecuencia de inferencia, presupuesto mensual, escalabilidad, experiencia técnica del equipo y número de features.

B

Desde un archivo YAML — cargar configuración

Descarga la plantilla con el botón ⬇️ Descargar plantilla, completa los campos en un editor de texto y sube el archivo con el botón Subir YAML. El formulario se llena automáticamente.

C

Usar un caso de ejemplo

Haz clic en uno de los tres botones de la sección Casos de ejemplo: 💳 Fintech (crédito, baja frecuencia), 🏦 Banco (batch nocturno), o 🚨 Fraude (tiempo real continuo). Los parámetros se llenan automáticamente.

Paso 2 — Ejecuta el análisis

Haz clic en el botón naranja 🔍 Analizar arquitectura en la parte inferior del panel lateral. El motor evalúa los 5 patrones en menos de 1 segundo y carga los resultados en la pantalla principal.

Nota: Si cambias algún parámetro después del análisis, debes volver a hacer clic en el botón para actualizar los resultados. El análisis no se actualiza automáticamente.

Paso 3 — Explora la recomendación

La pestaña 📊 Recomendación muestra los resultados organizados en dos áreas:

🏆
Panel de detalle (arriba)

Muestra la arquitectura seleccionada con su score total (anillo SVG animado), justificación narrativa, criterio decisivo, nivel de confianza, barras de desglose por criterio y chips de servicios AWS requeridos.

Por defecto muestra la arquitectura #1 (recomendada). Haz clic en Ver → en cualquier fila del ranking para ver el detalle de cualquier otra arquitectura.

📋
Ranking completo (abajo)

Muestra las 5 arquitecturas ordenadas por score, con los scores individuales por criterio (Lat, Frec, Pres, Esc, Exp) y el anillo de score total. La arquitectura actualmente visualizada muestra ✓ Viendo; las demás muestran Ver →.

Tip: Si aparece una advertencia de restricción dura (🚨), significa que esa arquitectura es técnicamente incompatible con tus parámetros. El motor la descarta automáticamente del ranking efectivo.

Paso 4 — Analiza la viabilidad con tus servicios

La pestaña 🔍 Análisis de servicios responde la pregunta: ¿Puedo implementar esta arquitectura con los servicios AWS que ya tiene mi empresa?

1

Selecciona la arquitectura a evaluar

El selector muestra automáticamente la arquitectura que tenías seleccionada en la pestaña anterior. Puedes cambiarlo para evaluar cualquiera de las 5.

2

Consulta el glosario de servicios

Abre el expander 📖 Glosario para ver qué hace cada servicio relevante para la arquitectura seleccionada, su categoría y su rol en el despliegue de ML.

3

Marca los servicios disponibles

Señala todos los servicios AWS que tu organización tiene contratados o autorizados. Los servicios se marcan con 🔵 (requerido), 🟡 (sustituto posible) o ⚪ (otros).

4

Haz clic en "Analizar viabilidad"

El sistema determina si puedes implementar la arquitectura con lo que tienes. Si falta algún servicio, propone sustitutos. Si no hay viabilidad, sugiere la alternativa más económica disponible.

Cargar configuración con YAML

El formato YAML permite guardar y compartir configuraciones de casos de uso, automatizar evaluaciones y mantener un historial reproducible de decisiones.

Estructura del archivo

modelo: nombre: "nombre-descriptivo" descripcion: "Qué hace el modelo" # texto libre framework: "sklearn" # sklearn · pytorch · tensorflow · xgboost · lightgbm · catboost · huggingface tipo: "clasificacion_binaria" # ver valores válidos abajo sector: "financiero" # opcional num_features: 10 # número entero despliegue: latencia_requerida_ms: 800 # milisegundos (50 – 86 400 000) frecuencia_inferencia: "baja" # baja · media · alta · continua presupuesto_mensual_usd: 50 # USD/mes (5 – 10 000) escalabilidad_requerida: "media"# baja · media · alta volumen_datos_kb: # opcional — se calcula si se omite organizacion: experiencia_tecnica: "media" # baja · media · alta proveedor_cloud: "aws"

Flujo de uso

  1. Haz clic en ⬇️ Descargar plantilla en el panel lateral.
  2. Abre el archivo en cualquier editor de texto (VS Code, Notepad, etc.).
  3. Modifica los valores según tu caso. Los campos marcados como opcional pueden dejarse en blanco.
  4. Guarda el archivo con extensión .yaml o .yml.
  5. En la aplicación, haz clic en el área de carga y selecciona el archivo.
  6. El formulario se llena automáticamente. Aparece un mensaje verde de confirmación con cuántos campos se cargaron.
Campos con error: Si un valor no es válido (ej: frecuencia_inferencia: "rapida"), ese campo se omite y los demás se cargan normalmente. Un mensaje rojo indica si no se reconoció ningún campo.

Lo que la herramienta NO puede hacer

❌ No crea la infraestructura

✗
No provisiona recursos en AWS. Solo genera el código y los pasos para que tú lo hagas.
✗
No ejecuta comandos Terraform, CDK ni CloudFormation de forma automática.
✗
No crea roles IAM, buckets S3 ni funciones Lambda en tu cuenta.

❌ No evalúa tu modelo específico

✗
No analiza el archivo del modelo (pickle, .pt, joblib). Solo trabaja con los parámetros de despliegue.
✗
No mide la latencia real de inferencia de tu modelo concreto.
✗
No estima el consumo de memoria o CPU real según el tamaño del modelo.

❌ No soporta todos los proveedores cloud

✗
No genera recomendaciones para GCP (Google Cloud) ni Azure. Solo AWS.
✗
No evalúa infraestructura on-premise ni clouds privadas.
✗
No considera servicios de Kubernetes gestionado (EKS) en esta versión.

❌ Limitaciones del motor

✗
Los costos mostrados son referenciales (us-east-1). Pueden variar por región, negociación o cambios de precios de AWS.
✗
No predice costos exactos: solo rangos y mínimos de referencia.
✗
No tiene en cuenta restricciones de compliance o regulatorias específicas de tu organización (GDPR, PCI-DSS, etc.).
✗
No reemplaza a un arquitecto cloud certificado para proyectos de alta criticidad.
Importante: Esta herramienta es un sistema de apoyo a la decisión, no una decisión final. Los scores son orientativos y están basados en una base de conocimiento codificada. Para proyectos de misión crítica o con requisitos regulatorios estrictos, complementa la recomendación con revisión técnica humana.

Cómo interpretar los resultados

Score total (anillo)

Rango 1–10. Es la suma ponderada de los 5 criterios. Un score de 8.0+ indica muy buena compatibilidad. Un score de 5.0–7.9 indica compatibilidad moderada con compromisos. Un score de <5.0 indica baja compatibilidad o restricciones activas.

Nivel de confianza

Porcentaje que refleja la diferencia entre el score del #1 y el #2, normalizada. 90–100% = decisión clara. 60–89% = decisión con alternativas cercanas. <60% = caso límite, considera revisar los parámetros.

Restricciones duras (🚨)

Son incompatibilidades físicas, no preferencias. Si Batch aparece con restricción y latencia < 1 000 ms, no es una sugerencia: Batch es incompatible con ese requisito, sin excepciones.

Tensiones (⚠️)

Son alertas no bloqueantes. Indican que hay un conflicto entre parámetros que puede generar costos inesperados o complejidad técnica. No descartan la arquitectura pero requieren atención.

Colores de las barras de criterio

  • Verde (≥ 7.5/10) — El criterio es claramente favorable para esta arquitectura
  • Ámbar (4.5–7.4/10) — Compatibilidad parcial, con matices
  • Rojo (< 4.5/10) — El criterio es desfavorable o hay restricción activa

Preguntas frecuentes

¿Necesito saber de AWS para usar la herramienta?
No. La herramienta está diseñada para que un data scientist sin experiencia cloud pueda obtener una recomendación válida. Los parámetros del formulario usan el lenguaje del problema, no el lenguaje de la infraestructura. El glosario de servicios explica qué hace cada servicio AWS.
¿Los costos mostrados son exactos?
Son costos mínimos de referencia para la región us-east-1, basados en el tier más económico de cada servicio. Los costos reales dependen del volumen de tráfico, la región AWS, negociaciones de contratos enterprise y posibles descuentos por Reserved Instances o Savings Plans. Úsalos como punto de partida, no como presupuesto definitivo.
¿Qué hago si la herramienta recomienda una arquitectura que no puedo implementar?
Ve a la pestaña Análisis de servicios, marca los servicios que tienes disponibles y haz clic en Analizar viabilidad. El sistema buscará sustitutos o, si no hay viabilidad, sugerirá la alternativa más económica que sí puedas implementar con lo que tienes.
¿Puedo confiar en el código de las guías de implementación?
El código de las guías es un punto de partida funcional, no código listo para producción sin revisión. Cubre el flujo básico de inferencia. Deberás adaptarlo a tu modelo específico, agregar manejo de errores según tu contexto, y someterlo a revisión de seguridad antes de desplegarlo en producción.
¿La herramienta accede a mi cuenta AWS o al archivo de mi modelo?
No. La herramienta es completamente local: no hace ninguna llamada a AWS, no accede a archivos de tu sistema (excepto el YAML que tú subes), y no envía datos a servidores externos. Todo el procesamiento ocurre en tu navegador/servidor local.
¿Puedo evaluar más de 5 arquitecturas o agregar arquitecturas personalizadas?
La versión actual evalúa exactamente los 5 patrones soportados (Serverless, Batch, Streaming, Containers, SageMaker). Agregar arquitecturas nuevas requiere modificar el motor (src/ml_arch_recommender/scoring/engine.py) y las guías (src/ml_arch_recommender/demo.py). La arquitectura del código está diseñada para ser extensible.
¿Qué pasa si mis parámetros cambian después de elegir una arquitectura?
Modifica los parámetros en el panel lateral y haz clic de nuevo en 🔍 Analizar arquitectura. Los resultados se actualizan completamente. Esto es especialmente útil para análisis de sensibilidad: cambiar el presupuesto o la frecuencia y ver cómo se modifica el ranking.
¿La herramienta funciona para modelos de deep learning o solo para modelos clásicos?
Funciona para cualquier tipo de modelo. El tipo de framework (sklearn, pytorch, tensorflow, huggingface) se incluye en la configuración YAML pero el motor no penaliza ni favorece arquitecturas por el tipo de framework. Lo que sí importa es si el modelo requiere GPU para inferencia (usa EC2 con instancias GPU, disponible a través de la arquitectura Containers).
¿Necesitas más ayuda? Consulta la Referencia Técnica para detalles del motor de puntuación, o la Propuesta de Valor para compartir con stakeholders.
