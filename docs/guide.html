Enfoque del proyecto — qué hace y qué NO hace el agente

🎯 El agente es un recomendador de arquitecturas de despliegue

El agente no entrena modelos. Recibe como entrada la descripción de un modelo ya existente (en este caso, un modelo de riesgo crediticio ya entrenado con sus parámetros) y los requerimientos de despliegue (latencia, presupuesto, frecuencia, etc.). A partir de esos criterios, recomienda qué arquitectura AWS usar para desplegarlo: serverless, batch, streaming, contenedores, o SageMaker.

El caso de estudio de Credit Risk es simplemente el objeto de estudio — el modelo que se quiere desplegar. Sirve para demostrar que la recomendación del agente es coherente y el despliegue funciona.

✅ Qué hace el agente
✓ Recibe parámetros de un modelo ya entrenado
✓ Evalúa 5 arquitecturas con scoring multicriterio
✓ Recomienda la arquitectura más adecuada
✓ Justifica la decisión con scores por criterio
✓ Valida la recomendación llamando al endpoint real
✓ Retorna el resultado completo al usuario
❌ Qué NO hace el agente
✗ No entrena modelos de ML
✗ No evalúa la calidad del modelo (AUC, F1, etc.)
✗ No gestiona la infraestructura en producción
✗ No monitorea el modelo después del despliegue
✗ No recom. hiperparámetros ni arquitecturas de red
✗ No es un AutoML
📌 Analogía para la sustentación El agente es como un arquitecto de software: no escribe el código de tu aplicación, pero sí te dice si debes usar microservicios, monolito o serverless. El "código" aquí es el modelo de ML ya entrenado.

Flujo n8n — los 6 nodos en detalle

Qué hace cada nodo, qué recibe y qué produce
#NodoTipoRecibeProduce
1 Webhook Trigger Webhook POST del usuario con caso_negocio + datos_prueba El body JSON completo pasa al siguiente nodo
2 Validar Input Code Node El JSON del webhook Mismo JSON si válido; error 400 si faltan campos requeridos
3 Motor de Recomendación Code Node caso_negocio con los 6 criterios de despliegue La recomendación completa con ranking de 5 arquitecturas, scores y justificación
4 Llamar Endpoint AWS HTTP Request datos_prueba (los 5 features del modelo) Respuesta del Lambda: riesgo ALTO/BAJO + probabilidad
5 Consolidar Respuesta Code Node Output del nodo 3 (recomendación) + output del nodo 4 (inferencia) Un único JSON con todo: recomendación + resultado AWS + validación
6 Webhook Response Respond to Webhook El JSON consolidado del nodo 5 HTTP 200 + JSON final al cliente
🔑 El nodo crítico es el 3 (Motor de Recomendación) Es el corazón del agente. Contiene toda la lógica multicriterio: tablas de scoring por arquitectura, pesos de cada criterio, y el cálculo del score ponderado. El código completo está en n8n/workflow_export.json y su equivalente en Python en src/ml_arch_recommender/scoring/engine.py.

🟡 Capa 2 — Base de conocimiento con razones documentadas (nodo 3)

Cada score del 0-10 tiene una razón trazable basada en comportamiento real de AWS. El agente puede explicar cualquier número que produce.

Batch + Latencia ≤1000ms → Score 1/10
"Batch es inherentemente asíncrono (minutos a horas). Incompatible con SLA de tiempo real. ← RESTRICCIÓN DURA"
Serverless + Frecuencia baja → Score 10/10
"Pay-per-request: $0 cuando no hay tráfico. Free tier AWS Lambda: 1M requests/mes. Ideal para uso esporádico."
SageMaker + Presupuesto $30/mes → Score 2/10
"ml.t3.medium: ~$38/mes siempre activo. Con presupuesto <$100 representa >38% del budget. ← COSTOSO"
Streaming + Frecuencia continua → Score 10/10
"Kinesis Data Streams: diseñado exactamente para procesamiento continuo de eventos en tiempo real."

🔴 Capa 3 — Detección de restricciones duras y tensiones

Detecta combinaciones de parámetros que un score ponderado no puede capturar solo. Se incluyen como advertencias[] en la respuesta aunque no cambien la recomendación.

coherencia Ganador requiere experiencia alta, equipo tiene baja "La arquitectura recomendada requiere experiencia media-alta. Planifica capacitación."
coherencia Payload < 1KB + frecuencia continua "El overhead de Kinesis puede superar el payload útil. Considera batching de micro-solicitudes."

Formato del modelo entrenado — cómo conectarlo

Qué espera el sistema, en qué formato y cómo sustituir con un modelo real

📌 El modelo viaja como artefacto serializado (.joblib) almacenado en S3

Lambda no tiene el modelo "adentro". En cada cold start, Lambda descarga el archivo .joblib de S3, lo carga en memoria, y lo reutiliza en todas las invocaciones siguientes (warm). El artefacto es un scikit-learn Pipeline que incluye el preprocesador y el clasificador en un solo objeto.

El artefacto DEBE ser un Pipeline de scikit-learn:
pythonEstructura mínima requerida
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler

import joblib

# El pipeline DEBE tener estos dos métodos:
#   pipeline.predict(X)        → array [0, 1, ...]
#   pipeline.predict_proba(X)  → array [[p0, p1], ...]

pipeline = Pipeline([
    ("scaler", StandardScaler()),
    ("classifier", LogisticRegression()),
])
pipeline.fit(X_train, y_train)

# Serializar — este archivo va a S3
joblib.dump(pipeline, "credit_model.joblib")
Qué puede cambiar sin modificar el Lambda:
✓ Puede cambiarEl algoritmo interno (RF, XGBoost, SVM, etc.)
✓ Puede cambiarEl preprocesador (MinMaxScaler, etc.)
✓ Puede cambiarEl número de features (actualizando FEATURE_ORDER)
✓ Puede cambiarEl nombre del archivo en S3 (variable env)
✗ No puede cambiarLa interfaz: debe tener predict_proba()
✗ No puede cambiarEl formato de salida: clase 1 = riesgo alto
⚠️ Algoritmos fuera de sklearn Si usas XGBoost, LightGBM o CatBoost, instálalos en el Lambda Layer. El pipeline de sklearn puede envolver estos modelos.

Cómo ejecutar n8n
3 formas de tenerlo corriendo — local, Docker o cloud gratuito
¿Qué es n8n? n8n es el orquestador de workflows que actúa como cerebro del agente. Recibe el JSON con las características del caso de uso, ejecuta el motor de recomendación en un nodo JavaScript, llama al Lambda AWS si lo necesitas, y devuelve la recomendación estructurada. El agente vive dentro de n8n — los demás componentes son solo infraestructura.
Opción 1 — n8n Cloud (recomendado para la demo del jurado)

Gratis para 1 workflow activo · No necesitas Docker · Accesible desde cualquier dispositivo · URL pública desde el primer día

1
Crear cuenta gratuita

Ve a app.n8n.cloud → "Get started for free" → Regístrate con Google o email.

2
Importar el workflow del proyecto

En n8n Cloud: menú izquierdo → Workflows → Import from file → selecciona n8n/workflow_export.json

3
Configurar variables de entorno

Settings → Variables → Agregar:

env
API_ENDPOINT  = https://tu-api-id.execute-api.us-east-1.amazonaws.com/prod/predict
4
Activar el workflow y copiar la URL del webhook

Haz clic en Activate (botón verde). El nodo "Recepción de Caso" mostrará su URL pública — esa es la dirección que llamas con POST.

Opción 2 — Docker (entorno local controlado)

Ideal durante desarrollo · Sin límite de workflows · Datos en tu máquina · Requiere Docker Desktop instalado

  --name n8n \

# Acceder: http://localhost:5678
# Usuario: admin  |  Contraseña: demo2026
Nota: El flag -v ~/.n8n:/home/node/.n8n persiste tus workflows y credenciales aunque el contenedor se detenga. Sin este flag, perderías todo al reiniciar.
bash

docker stop n8n
docker start n8n

# Ver logs en tiempo real
docker logs -f n8n
Opción 3 — npx (sin instalación, solo Node.js)

Sin Docker · Más fácil de arrancar · Solo para pruebas puntuales (no persiste datos por defecto)

Preguntas frecuentes del jurado — respuestas preparadas
¿Por qué n8n y no un microservicio Python directo?

n8n permite visualizar el flujo de decisión sin código. Un jurado no técnico puede ver el workflow y entender cada paso. Además, la integración con APIs externas (Lambda AWS) y con otras herramientas (Slack, Notion, Sheets) es nativa — el motor de scoring Python es reutilizable por separado.

¿El modelo de crédito es real?

Es sintético con reglas de negocio coherentes (ratio deuda > 65%, ingresos < 1.5M COP → alto riesgo). El foco del proyecto es el agente recomendador, no el modelo. Cualquier sklearn Pipeline con predict_proba() puede sustituirlo sin tocar el agente.

¿Cómo garantizan que la recomendación es correcta?

23 tests unitarios en 3 capas: (1) correctitud de la recomendación en escenarios conocidos, (2) calidad del razonamiento — cada criterio tiene una razón documentada de más de 10 caracteres, (3) detección de restricciones duras — batch no puede recomendarse cuando la latencia requerida es ≤1s.

¿Pueden extender esto a GCP o Azure?

Sí. El motor de scoring es agnóstico — define ArchitectureId como Literal y agrega entradas en la base de conocimiento _kb_*. Los equivalentes GCP serían Cloud Run (serverless), Dataflow (batch), Pub/Sub + Cloud Run (streaming), GKE (containers), Vertex AI (SageMaker equivalente).

57 servicios · 6 vectores por servicio · 3 capas de enriquecimiento sin alterar el scoring

Feature 1 — Enriquecimiento de restricciones

Cuando el motor detecta una restricción dura (es_restriccion=True) o un score ≤ 3 en un criterio, el campo nota_catalogo del CriterionDetail se puebla automáticamente con el texto constraints del catálogo para el servicio principal de esa arquitectura.

Ejemplo: batch con latencia ≤ 1s

python
batch.razonamiento["latencia"]
  score        = 1
  es_restriccion = True
  razon        = "Batch es inherentemente asíncrono
                  (minutos a horas)..."
  nota_catalogo = "[AWS Glue] Para consultas ad hoc
                   simples sobre S3, Athena puede ser
                   una opción más directa."

Ejemplo: streaming con frecuencia baja

python
streaming.razonamiento["frecuencia"]
  score        = 2
  es_restriccion = True
  razon        = "Montar infraestructura Kinesis
                  para uso esporádico es costoso..."
  nota_catalogo = "[Amazon Kinesis] No es la opción
                   más simple para cargas batch..."
El campo razon nunca se modifica — el catálogo agrega información en nota_catalogo sin sobrescribir el razonamiento del motor.

Por qué no se usan los 57 servicios del catálogo

El catálogo incluye servicios como WorkSpaces, Pinpoint, SES, GuardDuty, DMS, FSx y otros que no tienen relación con el despliegue de modelos ML. Integrarlos al motor introduciría ruido. Solo los 10–12 servicios directamente relevantes para los 5 patrones de arquitectura están mapeados. Los demás están disponibles en el CSV para futuros casos de uso (ej: si se extiende el agente a recomendar también pipelines de datos o seguridad).
