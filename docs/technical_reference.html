Visión general del sistema

El ML Arch Recommender es un sistema experto basado en reglas ponderadas que evalúa cinco patrones de despliegue de modelos Machine Learning en AWS y devuelve la arquitectura óptima junto con justificación, advertencias y guía de implementación lista para producción.

Flujo de datos

1
Entrada — El usuario proporciona 8 parámetros: descripción, tipo de modelo, latencia, frecuencia de inferencia, volumen de datos, presupuesto, escalabilidad y experiencia técnica.
2
Scoring — El motor evalúa los 5 patrones de despliegue contra cada criterio usando una base de conocimiento codificada con umbrales, scores 1–10 y flags de restricción dura.
3
Ponderación adaptativa — Los pesos base se ajustan según el perfil de entrada (presupuesto muy bajo, equipo sin experiencia) y se normalizan a suma = 1.0.
4
Ranking — Se calcula el score total ponderado para cada arquitectura y se ordena de mayor a menor. Las restricciones duras fuerzan el score a 1/10.
5
Salida — Se devuelve un RecommendationResult con ganador, ranking completo, justificación narrativa, confianza y advertencias detectadas.

CaseInput — estructura de entrada

Todos los parámetros de entrada se encapsulan en un dataclass inmutable. El motor nunca modifica el objeto recibido.

@dataclass
class CaseInput:
    descripcion:            str              # texto libre descriptivo del caso
    tipo_modelo:            str              # "clasificacion_binaria" | "propensity_scoring" | ...
    latencia_requerida_ms:  float            # p.ej. 200 (fraude) · 800 (app) · 86_400_000 (batch)
    frecuencia_inferencia:  FrequencyLevel   # "baja" | "media" | "alta" | "continua"
    volumen_datos_kb:       float            # KB por solicitud = num_features × 8 / 1024
    presupuesto_mensual_usd:float            # USD/mes máximo de infraestructura AWS
    escalabilidad_requerida:ScaleLevel       # "baja" | "media" | "alta"
    experiencia_tecnica:    ExperienceLevel  # "baja" | "media" | "alta"
    disponibilidad_requerida: str = "alta"   # reservado para versiones futuras
    servicios_disponibles: list[str] = field(default_factory=list)

Pesos base de los criterios

El score total de cada arquitectura es la suma ponderada de sus scores individuales en los 5 criterios. La fórmula base es:

score_total = Σ ( wi × scorei )   con   Σ wi = 1.0   y   scorei ∈ [1, 10]

Pesos adaptativos

El motor detecta dos perfiles especiales y redistribuye los pesos antes de calcular el ranking. Los nuevos pesos siempre se normalizan a 1.0.

Condición detectadaAjusteRazón
presupuesto ≤ $50 presupuesto → 0.35  |  latencia → 0.20  |  frecuencia → 0.20
escalabilidad → 0.10		 Para startups/MVP el costo es el factor más crítico; latencia y escala son secundarias
experiencia = "baja" experiencia +0.10  |  latencia −0.05 (mín 0.10)  |  escalabilidad −0.05 (mín 0.05) Un equipo sin experiencia técnica elige por facilidad de implementación primero
Nota: Ambos ajustes son acumulativos. Si presupuesto ≤ $50 Y experiencia = "baja" se aplican en secuencia y luego se normaliza. Los pesos finales pueden diferir significativamente de los base.

Matrices de puntuación por criterio

Cada función _kb_* en engine.py implementa la base de conocimiento de un criterio. Retorna un KBResult(score, justificacion, es_restriccion). Las restricciones fuerzan el score a 1 independientemente del peso.

Restricciones duras y advertencias

El motor detecta situaciones de incompatibilidad mediante la función _detect_advertencias(). Las restricciones duras son incompatibilidades físicas; las tensiones son señales de alerta no bloqueantes.

✗ HARD — Batch + tiempo real: Si latencia ≤ 1 000 ms, Batch Processing recibe score forzado a 1/10. “Batch es inherentemente asíncrono (minutos a horas). Incompatible con SLA de tiempo real.”
✗ HARD — Streaming + frecuencia baja: Si frecuencia = "baja", Streaming recibe score 2/10 con flag de restricción. “Montar infraestructura Kinesis es costoso e innecesario para tráfico esporádico.”
✗ HARD — Presupuesto insuficiente: Si el costo mínimo de la arquitectura supera el presupuesto disponible, se activa restricción presupuestal. SageMaker ($38/mes) queda bloqueado con presupuesto ≤ $200; Streaming ($11) y Containers ($15) con ≤ $50.

Estructura de salida — RecommendationResult

@dataclass
class RecommendationResult:
    recomendacion:    ArchitectureScore       # arquitectura ganadora
    ranking:          list[ArchitectureScore]  # las 5, ordenadas por score
    justificacion:    str                    # texto narrativo del por qué
    criterio_decisivo:str                    # criterio con mayor diferencial
    confianza:        float                  # diferencia score_1 - score_2 normalizada [0,1]
    advertencias:     list[Advertencia]      # restricciones y tensiones detectadas
    input_recibido:   CaseInput              # copia del input original

@dataclass
class ArchitectureScore:
    id:            str           # "serverless" | "batch" | "streaming" | "containers" | "sagemaker"
    nombre:        str           # nombre legible
    score_total:   float         # score ponderado final [1, 10]
    desglose:      dict[str,float]  # score por criterio
    posicion:      int           # posición en el ranking (1 = mejor)
    descartada_por:str | None    # razón si tiene restricción activa
    servicios_aws: list[str]    # servicios requeridos para esta arquitectura

⚡ Serverless (Lambda + API Gateway)

Costo: $0–$1/mes Latencia: 50–200 ms warm Cold start: mitigable

Patrón ideal para predicciones bajo demanda y baja/media frecuencia. El modelo se carga desde S3 al arrancar el contenedor Lambda y permanece en memoria mientras hay tráfico. El primer request tiene cold start (~500 ms); los siguientes son warm (~50–200 ms). Provisioned Concurrency elimina el cold start a costo fijo.

Servicios principales: AWS Lambda · Amazon API Gateway · Amazon S3
Cuándo no usar: frecuencia continua (timeout 15 min), volumen muy alto (Provisioned Concurrency costoso), o cuando se necesita GPU.

📦 Batch Processing (S3 + Glue)

Costo: $0.50–$50/mes Latencia: horas/días Requiere PySpark básico

Diseñado para scoring masivo diferido: campañas de marketing, rescoring nocturno de carteras, generación de listas de propensión. Un job Glue procesa millones de registros de S3 y escribe los resultados de vuelta a S3 o a una base de datos. EventBridge o Step Functions actúan como orquestador.

Servicios principales: AWS Glue · Amazon S3 · Amazon EventBridge · AWS Step Functions
Cuándo no usar: si se necesita respuesta en tiempo real (< 1 s). Esta es una restricción dura del motor.

🌊 Streaming (Kinesis + Lambda)

Costo: $11–$100/mes Latencia: ~200 ms Alta experiencia requerida

Arquitectura para procesamiento de eventos en tiempo real: cada transacción/evento llega a Kinesis Data Streams, dispara una Lambda que carga el modelo y genera la predicción, y almacena el resultado en DynamoDB para consulta inmediata. Escala horizontalmente via shards.

Servicios principales: Amazon Kinesis · AWS Lambda · Amazon DynamoDB · Amazon CloudWatch
Cuándo no usar: frecuencia baja (costo de Kinesis no se amortiza), presupuesto < $11/mes, equipo sin experiencia en arquitecturas event-driven.

🐳 Containers (ECS Fargate + ALB)

Costo: $15–$150/mes Latencia: 50–300 ms Requiere Docker + DevOps

API REST persistente sobre FastAPI o Flask containerizada en Docker, desplegada en ECS Fargate con ALB para balanceo. El modelo permanece en memoria sin re-cargas. Máximo control sobre el entorno de ejecución. Soporta blue/green deploys, rollback inmediato y auto-scaling por CPU/latencia.

Servicios principales: Amazon ECS · AWS Fargate · AWS App Runner · Amazon CloudWatch
Cuándo no usar: equipos sin experiencia en Docker/ECS, presupuesto < $15/mes (costo mínimo de Fargate).

🧠 SageMaker Managed Endpoint

Costo: $38–$500/mes Latencia: 30–150 ms warm Requiere MLOps maduro

Plataforma MLOps completa: el modelo se registra en el Model Registry, se despliega en un endpoint gestionado con auto-scaling nativo, monitoreo de data drift y model quality integrado, y A/B testing entre variantes sin downtime. Ideal cuando el ciclo de vida del modelo requiere re-entrenamiento frecuente y trazabilidad.

Servicios principales: Amazon SageMaker AI · Amazon S3 · Amazon CloudWatch
Cuándo no usar: presupuesto < $200/mes (ml.t3.medium ~$38/mes sin tráfico), proyectos simples donde el overhead operativo no se justifica.

La UI es un único archivo app.py (~1 700 líneas) organizado en secciones funcionales:

FunciónResponsabilidad

YAML API — formato de configuración

El sistema acepta configuración en formato YAML con tres secciones. El campo volumen_datos_kb es opcional y se calcula como num_features × 8 / 1024 si se omite.

modelo:
  nombre:             "mi-modelo-ml"
  descripcion:        "Texto descriptivo del caso de uso"
  framework:          "sklearn"           # sklearn|pytorch|tensorflow|xgboost|lightgbm|catboost|huggingface
  tipo:               "clasificacion_binaria"  # ver lista completa abajo
  sector:             "financiero"        # opcional
  num_features:       10

despliegue:
  latencia_requerida_ms:   800
  frecuencia_inferencia:   "baja"      # baja|media|alta|continua
  presupuesto_mensual_usd: 50
  escalabilidad_requerida: "media"     # baja|media|alta
  volumen_datos_kb:        # opcional

organizacion:
  experiencia_tecnica: "media"         # baja|media|alta
  proveedor_cloud:     "aws"           # solo aws soportado actualmente

Tipos de modelo válidos: clasificacion_binaria · propensity_scoring · regression · clustering · deteccion_fraude · scoring_credito · churn · aml
